# bug-fix-with-architecture-discipline

## 何时使用
- 修这个 bug
- 这个问题怎么修
- 感觉架构不太对
- 不要打补丁
- 这个 bug 很复杂
- 同类问题反复出现
- 改一处会影响很多地方
- 复现链路很长
- 上下文快不够了
- 需要先观察再定位
- 需要记录链路监控结果

## 核心原则

复杂 Bug 不直接进入 patch。

尤其是链路很长、上下文很多、现象跨多个模块时，不要让 AI 一直在大量上下文里反复猜测。

正确顺序是：

1. 明确现象
2. 制定定位计划
3. 建立观察点
4. 完善观察日志
5. 输出链路监控结果
6. 基于监控结果定位责任层
7. 锁定根因
8. 再决定修复点

## 工作流

### 1. Bug 现象重述
明确：
- 触发条件
- 复现路径
- 期望行为
- 实际行为
- 已知影响范围
- 是否属于长链路问题
- 是否存在上下文压力

### 2. 定位计划
先不要急着改代码，先输出：
1. 最可能根因（3~5 个，按概率排序）
2. 每个根因的验证方法
3. 优先检查的代码点 / 状态点 / 数据点
4. 最小定位路径
5. 是否需要增加观察日志
6. 是否需要先生成链路监控结果

### 3. 建立观测点
明确需要观察：
- 状态字段
- 函数入参 / 出参
- 渲染条件
- 数据转换结果
- 分支命中情况
- 异步时序
- 事件触发顺序
- 缓存命中情况
- 第一个变错的位置
- 哪一层最先偏离预期

### 4. 长链路观察日志

当工作链路较长、模块较多、上下文压力较大时，必须优先补观察日志，而不是继续扩大猜测范围。

观察日志应该服务于“压缩上下文”，而不是制造更多噪音。

#### 观察日志要求
每条日志至少包含：
- 链路节点名
- 当前阶段
- 关键输入
- 关键输出
- 关键状态
- 分支命中结果
- 与预期是否一致
- traceId / requestId / nodeId / fileId 等可串联标识

#### 推荐格式

```text
[Trace][链路节点]
traceId: xxx
stage: xxx
input: xxx
stateBefore: xxx
branch: xxx
output: xxx
stateAfter: xxx
expected: xxx
matched: true/false
```

#### 日志纪律
- 不要一次性打印整个大对象
- 优先打印判断所需的最小字段
- 每层只记录能证明责任归属的关键点
- 观察点应该覆盖链路入口、中间转换点、责任边界、最终输出点
- 发现第一个偏离点后，应停止扩大范围，转入责任层判断

### 5. 链路监控结果

当观察日志收集完成后，先输出“链路监控结果”，再继续定位。

链路监控结果是对大量日志和上下文的压缩摘要，用来避免上下文上限导致定位崩盘。

#### 输出模板

```text
## 链路监控结果

### 1. 链路路径
入口 -> 模块A -> 模块B -> 模块C -> 输出

### 2. 正常节点
- 节点A：输入正确，输出正确
- 节点B：状态转换正确

### 3. 异常节点
- 节点C：输入正确，输出错误

### 4. 第一个偏离点
节点C / 方法名 / 文件位置

### 5. 责任层判断
数据层 / 状态层 / 业务逻辑层 / 渲染层 / 架构边界层

### 6. 排除项
- 已排除模块A，因为输入输出一致
- 已排除模块B，因为状态未偏离

### 7. 下一步定位动作
只检查节点C附近逻辑，不继续扩大范围
```

### 6. 责任层判断
先判断问题首先出现在哪一层：
- 数据层
- 状态层
- 业务逻辑层
- 渲染层
- 架构边界层

判断依据必须来自观察日志或链路监控结果，不要凭感觉跳到修改方案。

### 7. 修复决策
仅在根因基本锁定后，输出：
- 推荐修改点
- 不推荐修改点
- 修复方案
- 为什么这里是正确 owner
- 为什么不是调用方兜底
- 是否需要补测试

### 8. 架构升级判断
若根因来自职责不清、耦合过高、扩展点不足、边界错误：
1. 当前止血方案
2. 架构优化建议
3. 是否纳入技术债
4. 是否建议拆成后续治理任务

## 默认输出
1. Bug 理解
2. 定位计划
3. 观测点
4. 长链路观察日志建议（仅长链路时输出）
5. 链路监控结果（已有日志或上下文压力大时输出）
6. 根因与归属
7. 修复方案
8. 架构建议
9. 风险

## 禁止行为
- 根因未锁定就直接给 patch
- 日志没有第一个偏离点就扩大修改范围
- 用调用方兜底掩盖底层问题
- 把长链路观察能力拆成独立 skill
- 把链路监控结果当成项目管理日报
