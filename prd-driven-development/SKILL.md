# prd-driven-development

## 何时使用
- 新功能定义
- 交互规则调整
- 业务规则变化
- 协议语义变化
- 验收标准确认
- 需要把模糊需求收敛成明确规则

## 不主导场景
- 纯内部重构
- 性能优化（不改行为）
- 小 bug 修复（不改产品规则）
- 只调整实现细节、不影响用户可感知行为

## 工作原则
1. 先对齐 PRD，再实现代码。
2. 若 PRD 已覆盖，则按 PRD 实现。
3. 若需求与 PRD 冲突，先更新 PRD。
4. 若多文档冲突，先收敛为一致版本。
5. 若 PRD 变更影响模块边界、数据流、职责划分或性能模型，要提醒做架构检查。

## 每轮输出
1. 是否需要更新 PRD
2. 修改了哪些定义
3. 影响范围
4. 是否需要补测
5. 实现建议
